01

How to use this

Every program starts from the same universal base, then adds one LoB theme on top. Nothing else is prescribed.

The base — always the same

Ink hierarchy, Thumbtack accent blue, cards, rules, buttons, typography, feedback states. These don't change program to program. If you find yourself redesigning ink colors or accent blue, stop — that's the base's job.

The LoB theme — one per program

Each program picks the palette of the line of business it serves. Sales gets green; Success gets indigo; Service gets purple; Trust & Safety gets yellow. That palette shows up in program headers, tabs, section markers, and identity moments — not everywhere.

The one rule: LoB color is identity. Accent blue is decisions. Don't swap them. A learner clicks accent blue to advance, and reads LoB color as "this is a Sales program" or "this is a Trust & Safety program." If accent blue starts drifting into LoB territory (or the reverse), the signal breaks.

Two typefaces cover almost everything. Inter for HTML and web embeds; Rise inside Compass (Sana Labs) native cards, since Rise is what Compass renders. Caveat is a third option for handwritten annotations — use sparingly.

When to reach for Caveat: pedagogical marginalia — a facilitator's aside, a Post-it-style prompt, a handwritten cue. Not for body copy, buttons, or anything the learner has to read at length. Caveat is a texture, not a workhorse.

Layout & surfaces

Content sits on white cards separated by hairline rules. Rounded corners at 8–12px. Shadows are subtle — the system should feel calm, not hovering.

Card anatomy

Card eyebrow
A card title
This is the card body. Cards can also contain lists, actions, or media — they're just the base container the system leans on.

Spacing scale

An 8-point grid with named tokens. Padding inside cards usually lands at 20–28px. Gaps between cards in a stack usually land at 12–20px. Section-to-section vertical rhythm sits at 48–64px.

  • 4Tight spacing — icon-to-label gaps, chip padding
  • 8Small gaps — between adjacent chips or lines within a card
  • 16Default gap — between related cards
  • 24Card interior padding — comfortable breathing room
  • 48Section-to-section vertical rhythm

Components

Buttons, chips, and pills — the small workhorses. Everything more complex is built from these plus a card.

Buttons

Primary for the decision-forward action on the screen. Secondary (ink) for significant secondary actions that shouldn't compete with the primary. Quiet for tertiary or reset-type actions. Text for inline navigation.

Chips & pills

Small classifier or status markers. Neutral by default; accent for "in-system" tags; warn for attention; good for confirmation.

Feedback states

Left-border stripe on a white card. Consistent across positive, warning, and decision moments. No panel fills — feedback panels shouldn't take over the reader's attention field.

Nicely done. The learner captured what was asked for. Move on when ready.
One slot is still empty. Look back at the transcript for what they said about that dimension.
Recommendation: A decision-moment panel. Neutral in-system tone, accent-blue anchor.
Why border-only, not filled? A green or yellow-filled panel drives attention hard and can feel alarming or celebratory in a way that competes with the copy. A hairline card with a colored left-stripe carries the same semantic signal at a fraction of the visual weight.
08

Icons

Two kinds of icons in the system: UI icons are functional interface furniture; concept icons are program-specific pedagogical anchors. Different weight, different rules.

UI icons — Lucide as the default

For HTML interactive builds, use Lucide. It's an open-source line-art icon library, ~1400 icons, MIT-licensed, on a 24×24 grid with 2px stroke and currentColor. It maps closely to Thumbtack's own icon-pack aesthetic and Claude knows it natively.

Two ways to include them

Inline SVG — best for artifact portability (works when the file is opened offline, embedded, or shared):

<!-- Arrow right --> <svg width="20" height="20" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round"> <path d="M5 12h14"/> <path d="m12 5 7 7-7 7"/> </svg>

CDN — best for larger builds where you'd otherwise repeat many inline SVGs:

<script src="https://unpkg.com/lucide@latest/dist/umd/lucide.min.js"></script> <i data-lucide="arrow-right"></i> <script>lucide.createIcons();</script>

Rules for UI icons

  • Size16–20px inline; 24px inside buttons; 32–40px for feature callouts. Don't scale beyond 40px — that's concept-icon territory.
  • Stroke2px. Lucide's default. Don't customize per icon.
  • ColorcurrentColor — inherits from the surrounding text. Never hardcode a hex on a UI icon.
  • FillLine-art only in this system. No filled icons — they read as a different visual language.
  • SignalAn icon that doesn't add signal is noise. If a button label says "Save," it doesn't need a floppy-disk icon next to it. Ask what the icon adds; if the answer is "nothing," remove it.
On the Thumbtack icon pack. Thumbtack's own icon pack (delivered as an Adobe XD file) stores icons in Adobe's proprietary format — SVGs aren't automatically extractable. When someone on the team opens the XD file and exports icons to SVG, they slot in as drop-in replacements using the same conventions (24×24 grid, currentColor). Until then, Lucide covers the same visual territory.

Concept icons — one set per program

Concept icons are program-specific pedagogical anchors. Each represents a big idea the program teaches — a reading-glasses icon for a "read the customer" concept, a signpost for "choose a path," a waveform for "read the live signal." They're developed once per program by the program lead, alongside the pedagogy itself.

The key difference from UI icons: concept icons carry meaning. They belong at title pages, module intros, and hero moments — not on buttons or in table headers. If they show up as UI, the pedagogical weight dissipates.

Style spec

  • StyleLine-art, single color (ink #2F3033 by default, or the program's LoB primary).
  • Stroke2.5px — slightly heavier than UI icons because these render larger.
  • GridviewBox 160×160 (standard) or 200×200 (for more complex compositions).
  • Capsstroke-linecap="round", stroke-linejoin="round" for the sketched quality.
  • AccentsOne accent color allowed for pedagogical emphasis (e.g., a yellow highlight on the "chosen route" arm of a signpost). Reserved — use rarely.

Development process

Program lead identifies 3–8 anchor concepts in the curriculum. Each concept gets a visual metaphor — an object that carries the meaning, not just illustrates it. Reading glasses ≠ "read"; reading glasses = "examine carefully." Sketch first, then produce as SVG. Review the set together — do the icons read as siblings? Same line weight, same conceptual altitude?

Example — AE Upskilling program. Reading glasses (Business Context — examining static content), waveform (Engagement Context — attuning to live signal), signpost (Package Routing — choosing a direction), clipboard (Discovery — capturing what you hear). Each icon anchors a course; each was sketched during storyboard, refined alongside the pedagogy, produced as SVG when the concepts were locked. The set reads as siblings because they were designed as a set.

One rule about concept icons

Don't use them as UI. A signpost belongs on a title page; it doesn't belong as the icon on a "next" button. The whole point of concept icons is that they carry pedagogical weight — if they show up everywhere, the weight dissipates and they become decoration.

09

Composition rules

The small handful of things you shouldn't reinvent per program. Everything else — layout, illustration style, motion — is fair game.

  • 01Ink for text. Body copy is #2F3033. Don't tint text with LoB color even lightly — contrast will fail somewhere and legibility takes a hit.
  • 02Accent blue for decisions. When a learner is meant to click, choose, or advance, the target is accent blue. If accent blue starts appearing on identity moments (headers, tabs), the "click me" signal gets diluted.
  • 03One LoB per program. Sales green in a Sales program, indigo in a Success program. Don't mix families within a program — the identity read stops working.
  • 04LoB color for identity, not for actions. Program header, tabs, section markers, callouts — yes. Buttons, links, active states — no. Those stay accent blue.
  • 05Feedback stays border-only. Good, warn, and accent panels use a colored left-border on a white card. Not filled backgrounds.
  • 06Yellow reads as attention. If your program isn't Trust & Safety and you use yellow, learners will read T&S adjacency. If you need a yellow (like the AE program's captured-evidence highlight), document the local rule so it stays intentional.
10

Extending the system

The base doesn't cover everything. When your program needs something the base doesn't provide, extend rather than override.

Add, don't replace

If your program needs a mechanic the base doesn't have — a capture pattern, a scoring rubric, a specific card variant — add it as a program-specific rule on top of the base. Don't rewrite base tokens or component patterns; leave them alone so other programs still recognize them.

Document your locals

When you add a program-specific pattern, note it. What the pattern is, what it means, when to use it. This becomes the "reading the pro captured-evidence yellow" of the next program — unless it's documented, someone will trip over it later.

Field Notebook is an example, not a model

The AE Upskilling program's Field Notebook system is one instance of "base + LoB + program-specific pattern." It uses Sales-adjacent tones (though it currently leans on accent blue rather than sales green — a legacy choice), adds a yellow captured-evidence mechanic, and layers Caveat-font marginalia. That combination works for that program's mechanic. It's not the shape every program should take — it's the shape one program took.

The test: if a designer from another team opened your program and saw only the base + LoB parts, would it feel consistent with theirs? If yes, you extended correctly. If no, something has drifted into base territory and should probably come back out.
11

Using with Claude

Interactive builds usually happen in Claude. This system is designed to be pasted in as context so Claude produces output that matches the base without further instruction.

Start every build with the cheat sheet

The cheat sheet is the compact companion to this doc — the same system, distilled to what Claude actually needs. Paste it at the top of a new Claude conversation before describing what you want built. Claude will read the tokens, internalize the rules, and produce output that follows the system.

Where to find it: enablement-visual-cheatsheet.html — either upload the file to your Claude conversation, drag-drop it in, or copy-paste its contents.

A good prompt template

Building an interactive [artifact type] for [program name]. LoB: [Sales / Success / Service / Trust & Safety]. Following the Enablement Design Visual System Base. [paste cheat sheet contents here, or upload the file] Program-specific extensions: - [any custom patterns, e.g. "captured evidence highlight uses #FFDD80"] - [any concept icons used] Task: [describe what to build]

Why this works

Claude produces output based on what's in its context window. The cheat sheet gives it structured tokens, LoB palettes, and rules — enough to compose new components correctly. The LoB spec tells it which palette to activate for identity moments. Extensions tell it what the program does on top of the base. The task tells it what to build.

Iterating with Claude

Claude follows the tokens once loaded. You don't need to remind it on every message. If output starts drifting — a non-base color appears, or a rule breaks — point to the specific rule number. The composition rules in the cheat sheet are numbered for exactly this: "That violates rule 04 — LoB color shouldn't appear on buttons."

Document extensions as you add them. When you extend the system mid-build (adding a new capture pattern, a new card variant, a program-specific concept icon), tell Claude the extension explicitly and reference it going forward. Claude won't invent it consistently on its own — be the authority on your program's local rules.

The base takes precedence in conflicts. If Claude proposes a pattern that conflicts with the base, the base wins. If a program-specific extension conflicts with the base, the extension probably needs revising to fit.

Starter CSS

Copy the block below into any new program's root stylesheet. Uncomment the one LoB block your program uses.

/* ══════════════════════════════════════════════════════════ Enablement Design · Visual System Base Universal base + one LoB. Add program-specifics after. ══════════════════════════════════════════════════════════ */ :root { /* ── Base ── */ --ink: #2F3033; --ink-soft: #4B5563; --ink-muted: #6B7280; --ink-faint: #9CA3AF; --surface: #FFFFFF; --page: #FFFFFF; /* override per program if needed */ --rule: #E5E7EB; --rule-soft: #F3F4F6; /* Thumbtack accent — decisions, primary actions */ --accent: #009FD9; --accent-deep: #007FAD; --accent-soft: #B2EBFF; --accent-bg: #EAF6FA; /* Feedback */ --warn: #D97706; --warn-soft: #FEF3C7; --good: #059669; --good-soft: #D1FAE5; --good-deep: #065F46; /* ── LoB — uncomment the one your program uses ── */ /* Sales */ --lob: #2DB783; --lob-deep: #054E33; --lob-light: #DEF1EC; --lob-mid: #C6F7DA; /* Success --lob: #5968E2; --lob-deep: #383C80; --lob-light: #E8F1FD; --lob-mid: #CCE1FF; */ /* Service --lob: #8D56EB; --lob-deep: #4E2D86; --lob-light: #F6EEF6; --lob-mid: #DCCBE3; */ /* Trust & Safety --lob: #FDBE17; --lob-deep: #A77005; --lob-light: #FDF7E7; --lob-mid: #FFEBB3; */ } * { box-sizing: border-box; margin: 0; padding: 0; } body { background: var(--page); color: var(--ink); font-family: 'Inter', system-ui, sans-serif; line-height: 1.55; -webkit-font-smoothing: antialiased; }

That's the starting point. Every other file in your program builds on top of this — sections, cards, buttons, feedback panels — using the tokens above. Program-specific patterns (a capture mechanic, a scoring rubric, custom cards) go in a separate block below the base, clearly labeled.
